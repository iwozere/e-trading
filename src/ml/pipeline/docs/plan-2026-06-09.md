# ML Pipeline Consolidation & Bug-Fix Implementation Plan

## Context

The `src/ml/pipeline/` directory contains 19 pipelines (P00–P18) accumulated across research iterations.
The architecture review dated 2026-06-09 (`src/ml/pipeline/docs/review-2026-06-09.md`) identified:
- 3 critical bugs that inflate backtest Sharpe ratios vs live performance
- 4 near-duplicate pipeline pairs that can be merged without losing signal capability
- 2 pipelines to deprecate (P05 superseded, P11 empty)
- 1 pipeline to reclassify (P13 is a risk overlay, not a signal pipeline)

This plan fixes bugs in priority order, then merges duplicates into configurable unified pipelines, then adds the validation infrastructure needed before re-tuning.

## Phase 1 — Critical Bug Fixes (Week 1–2)

### 1.1 Fix triple-barrier label leakage — P07 evaluator

**File:** `src/ml/pipeline/p07_combined/evaluator.py`

**Problem:** `run_evaluation()` computes triple-barrier labels on the full OHLCV dataset, then performs a 70/30 split. Labels near the split boundary are computed using test-set prices. With `tpl_hours=96` at 15m resolution this contaminates up to 384 training samples.

**Fix — replace the current `run_evaluation()` body with a pre-split labeling approach:**

```python
# Step 1: calculate tpl_bars from tpl_hours
tpl_hours = params.get('tpl_hours', 12 * 15 / 60)
tpl_bars  = cls.hours_to_bars(tpl_hours, timeframe)
params_with_bars = {**params, 'tpl_bars': tpl_bars}

# Step 2: split raw OHLCV FIRST, leaving a buffer equal to tpl_bars
n = len(ohlcv)
train_end = int(n * 0.60) - tpl_bars      # 60% train, minus buffer
val_end   = int(n * 0.80) - tpl_bars      # next 20% val, minus buffer
# final 20% is true OOS test — never touched by Optuna

ohlcv_train = ohlcv.iloc[:train_end]
ohlcv_val   = ohlcv.iloc[train_end:val_end]
ohlcv_test  = ohlcv.iloc[val_end:]

# Step 3: compute features + labels INDEPENDENTLY on each segment
X_train, y_train = cls.prepare_data(ohlcv_train, params_with_bars)
X_val,   y_val   = cls.prepare_data(ohlcv_val,   params_with_bars)
X_test,  y_test  = cls.prepare_data(ohlcv_test,  params_with_bars)
```

The Optuna objective (`optimize.py`) trains on `X_train`/`y_train`, evaluates on `X_val`/`y_val`.
`save_artifacts()` reports final metrics on `X_test`/`y_test` only.

**Same fix applies identically to `src/ml/pipeline/p08_mtf/evaluator.py`** — copy the change verbatim since P08Evaluator extends P07Evaluator but overrides `prepare_data`.

**After fix:** invalidate and delete all existing Optuna study `.db` files — they were optimised against a contaminated validation set. Run new Optuna studies from scratch.

### 1.2 Fix blocking `input()` — P01 pipeline runner

**File:** `src/ml/pipeline/p01_hmm_lstm/run_pipeline.py`

**Problem (a):** Lines 266 and 282 call `input()` when stages fail, hanging the process in any non-interactive context (scheduler, Docker, CI).

**Problem (b):** The default `skip_stages = [1, 2]` on line 477 silently skips data loading and HMM training when no `--skip-stages` flag is given.

**Fix (a):** Add `--interactive` flag; default `False`. Replace both `input()` calls:
```python
# argparse addition
parser.add_argument('--interactive', action='store_true',
    help='Prompt before continuing after stage failures (default: non-interactive)')

# In run_pipeline(), replace input() with:
if self.interactive:
    response = input(f"\nCritical stage {stage_num} failed. Continue? (y/n): ")
    if response.lower() != 'y':
        break
else:
    if fail_fast:
        break   # non-interactive + fail_fast → stop immediately
    # else continue automatically
```

**Fix (b):** Change the default `skip_stages` on line 477:
```python
# BEFORE: skip_stages = [1, 2]  (bad default)
# AFTER:
skip_stages = []
```
Add a log warning if `--skip-stages` is passed with values 1 or 2 advising the user that skipping data loading/HMM training requires cached artefacts to exist.

### 1.3 Fix Optuna contamination — P07 / P08

**File:** `src/ml/pipeline/p07_combined/optimize.py`

**Problem:** Optuna's objective function calls `P07Evaluator.run_evaluation()` which (after fix 1.1) now returns separate train/val/test sets. Update the objective to train on train set and evaluate on val set only:

```python
def objective(trial, ohlcv_clean, timeframe="15m"):
    params = { ... }  # unchanged parameter suggestions

    res = P07Evaluator.run_evaluation(ohlcv_clean, params, timeframe=timeframe)
    if "error" in res:
        return -1.0

    # Use val-set portfolio for Optuna, never test-set
    sharpe = res["pf_val"].sharpe_ratio()
    total_trades = res["pf_val"].trades.count().sum()

    if pd.isna(sharpe) or total_trades < 20:
        return -1.0

    return float(sharpe * np.log10(max(total_trades, 2)))
```

`save_artifacts()` in `pipeline.py` uses `res["pf_test"]` (true OOS) for all reported metrics.

**Same change applies to `src/ml/pipeline/p08_mtf/optimize.py`.**

### 1.4 Fix P01 indicator-optimization contamination

**File:** `src/ml/pipeline/p01_hmm_lstm/x_04_optuna_indicators.py`

**Problem:** Stage 4 (indicator Optuna) computes its objective on 80% of the full dataset, but then validates against the full dataset (line 375). Stage 5 then receives a dataset whose indicators were already curve-fitted.

## Phase 2 — Screener Merge: P06 + P10 → configurable EMPS (Week 3–5)

### 2.1 Design: pluggable Stage 3 analyzer

Both P06 and P10 share identical Stages 1, 2, 4–8 (universe, fundamental filter, rolling memory, UOA, results, sentiment, alerts). Only Stage 3 differs: P06 uses `VolatilityFilter`, P10 uses `AccumulationAnalyzer`.

Define a protocol in `src/ml/pipeline/shared/analyzer_protocol.py`:
```python
from typing import Protocol, List
import pandas as pd

class ScreenerAnalyzer(Protocol):
    def apply_filters(self, tickers: List[str]) -> pd.DataFrame: ...
```

Both `VolatilityFilter` (shared) and `AccumulationAnalyzer` (from P10) already expose `apply_filters()` — no internal changes needed.

### 2.2 Changes to `src/ml/pipeline/p06_emps2/`

**`config.py`** — add `analyzer_type` to `EMPS2PipelineConfig`:
```python
analyzer_type: Literal["volatility", "accumulation"] = "volatility"
```
Also add EMPS3-style rolling memory fields (`lookback_days`, slope thresholds) so both rolling memory modes are configurable from one config class.

**`emps2_pipeline.py`** — Stage 3 becomes a factory:
```python
def _build_analyzer(self) -> ScreenerAnalyzer:
    if self.config.analyzer_type == "accumulation":
        from src.ml.pipeline.p06_emps2.accumulation_analyzer import AccumulationAnalyzer
        return AccumulationAnalyzer(self.config.filter_config, ...)
    return VolatilityFilter(self.data_manager, self.config.filter_config, ...)

def _stage3_analysis(self, tickers: list) -> pd.DataFrame:
    return self._build_analyzer().apply_filters(tickers)
```

**Move** `src/ml/pipeline/p10_emps3/accumulation_analyzer.py` → `src/ml/pipeline/p06_emps2/accumulation_analyzer.py`. Update all imports in P10.

## Phase 4 — Predictive Model Merge: P00 + P01 → multi-regime P01 (Week 7–9)

### 4.1 Design: `multi_regime` flag in P01

P00's contribution is training one LSTM per HMM-detected regime instead of one LSTM with regime as a feature column.

Add `multi_regime: bool` to `config/pipeline/p01.yaml`:
```yaml
lstm:
  multi_regime: false   # true → train separate LSTM per regime (P00 approach)
  min_regime_samples: 200
```

**`x_06_train_lstm.py`** — add branch:
```python
if config['lstm']['multi_regime']:
    _train_multi_regime(df_labeled, config)
else:
    _train_single(df_labeled, config)          # current behaviour

def _train_multi_regime(df, config):
    for regime_id in df['hmm_regime'].unique():
        regime_df = df[df['hmm_regime'] == regime_id].copy()
        if len(regime_df) < config['lstm']['min_regime_samples']:
            _logger.warning("Regime %d has only %d samples, skipping", regime_id, len(regime_df))
            continue
        _train_single(regime_df, config, suffix=f"_regime{regime_id}")
```

The per-regime LSTM filenames get a `_regime{id}` suffix so inference knows which model to load. **`x_07_validate_lstm.py`** must load per-regime models when `multi_regime=True`.

## Phase 6 — Validation Infrastructure & Mandatory Robustness (Week 11–12)

### 6.1 Mandatory robustness gate in P07 `save_artifacts()`

After the evaluator returns results, run `P07RobustnessChecker.run_all_checks()` and compute WFE:
```python
robustness = P07RobustnessChecker(ticker, timeframe, res_dir / "robustness")
rob_results = robustness.run_all_checks(ohlcv, params, res)
avg_oos_sharpe = rob_results['wfa'].get('avg_oos_sharpe', float('nan'))
is_sharpe = res['pf_test'].sharpe_ratio()
wfe = avg_oos_sharpe / is_sharpe if is_sharpe and is_sharpe > 0 else 0.0

_logger.info("Walk-Forward Efficiency: %.2f (threshold 0.50)", wfe)
if wfe < 0.5:
    _logger.warning("Strategy REJECTED for %s %s: WFE=%.2f < 0.50", ticker, timeframe, wfe)
    return False   # do NOT write completed.flag
```

WFE threshold of 0.50 means the OOS Sharpe must be at least half the IS Sharpe.

### 6.2 TRF cache TTL validation

**File:** `src/ml/pipeline/shared/trf_downloader.py`

After loading a cached TRF file, validate its age:
```python
def _is_cache_fresh(path: Path, max_age_days: int = 1) -> bool:
    if not path.exists():
        return False
    age = (datetime.now() - datetime.fromtimestamp(path.stat().st_mtime)).days
    return age <= max_age_days
```
Force refresh if cache is stale. Log the TRF file's data date alongside every volume metric in audit logs.

### 6.3 Rolling memory bootstrap health-check

**File:** `src/ml/pipeline/p06_emps2/rolling_memory.py`

Persist `first_run_date` and `cumulative_phase1_count` in a small JSON state file inside the results directory.

## Phase 8 — Re-Tuning Guide (manual runs after all fixes land)

Phase 8 is not automated — run Optuna studies manually after all code changes are in place. Follow this sequence.

### Step 1 — Clear stale Optuna databases

Old studies were optimised against the contaminated 70/30 split and are invalid. Archive them:

```bash
# from project root
mkdir -p archive/optuna_contaminated
mv src/ml/pipeline/p07_combined/optuna_study.db  archive/optuna_contaminated/p07_$(date +%Y%m%d).db
mv src/ml/pipeline/p08_mtf/optuna_study.db       archive/optuna_contaminated/p08_$(date +%Y%m%d).db
find src/ml/pipeline/p01_hmm_lstm/ -name "*.db" -exec mv {} archive/optuna_contaminated/ \;
```

### Step 2 — Re-tune P07 (with optional MTF)

Run base P07 first (MTF disabled). Only enable MTF if base WFE < 0.5.

```bash
# Base mode:
python src/ml/pipeline/p07_combined/pipeline.py --ticker NVDA --tf 15m --years 2022,2023,2024

# Check: results/p07_combined/NVDA/15m/.../robustness_summary.json
# wfa.avg_oos_sharpe >= 0.5 AND avg_oos_sharpe / IS_sharpe >= 0.5

# If WFE < 0.5, retry with MTF:
python src/ml/pipeline/p07_combined/pipeline.py --ticker NVDA --tf 15m --years 2022,2023,2024 --enable-mtf
```

**Acceptance per ticker:** OOS Sharpe ≥ 0.5, WFE ≥ 0.5, trade count ≥ 30. If all fail after MTF is enabled, the ticker/timeframe has no reliable edge — do not trade live.

### Step 3 — Re-tune P01 (HMM-LSTM, optionally multi-regime)

```bash
# Single-regime first:
python src/ml/pipeline/p01_hmm_lstm/run_pipeline.py --config config/pipeline/p01.yaml

# Check: results/p01_hmm_lstm/{symbol}/validation_report.json

# If accuracy < 55%, set lstm.multi_regime: true in p01.yaml and re-run.
```

Multi-regime training produces one `.pt` file per HMM state. Verify inference selects the right model based on current regime before going live.

### Step 4 — Re-tune P02 (CNN variant)



## Risk & Sequencing Notes

- **Phase 1 must complete before Phase 8** — tuning on contaminated data produces invalid models.
- **Phases 2–5 are independent of each other** — they can run in parallel.
- **P06 + P10 merge (Phase 2) is lowest risk** — both pipelines share the same shell; only Stage 3 changes.
- **P07 + P08 merge (Phase 3) is medium risk** — P08's MTF loader and features must be regression-tested against P07 baseline before deprecating P08.
- **P02 + P03 merge (Phase 5) is highest risk** — different intermediate storage formats (NPZ vs CSV) and different target strategies require careful data-flow testing before deleting P03.
- **Do not remove deprecated pipeline folders** until shim wrappers have been in production for at least 4 weeks and no scheduler references the old entrypoints.
