# ML Pipeline Architecture Review — 2026-06-09

**Scope:** All 19 pipelines in `src/ml/pipeline/` (P00–P18)  
**Reviewer:** Claude Sonnet 4.6  
**Purpose:** Input document for the pipeline refactoring project. Focus is on trading correctness, data leakage, validation quality, production readiness, and code-level improvements.

---

## 1. Executive Summary

The platform contains 19 ML/quant pipelines split across two clear families:

- **Screener/signal pipelines** (P04, P05, P06, P10, P12, P14, P17, P18): produce daily watchlists from rule-based or light-ML filters. No sequential training loop; signals come from structural market data.
- **Predictive model pipelines** (P00, P01, P02, P03, P07, P08, P09, P13, P16): train ML models (HMM, LSTM, CNN, XGBoost) and generate directional signals.

The screener family is largely production-ready. The predictive model family has several structural problems — the most severe being a **boundary data leakage** in P07 (and likely P08 which inherits the same evaluator), a **blocking `input()` call** in P01, and **missing walk-forward validation** across most ML pipelines. These issues would cause backtest Sharpe ratios to be inflated compared to live trading performance.

The shared `timeseries_split.py` module is well-designed and enforces correct split-before-scale semantics. However, **P07 bypasses it** and rolls its own 70/30 split directly in the evaluator, which introduces the boundary leakage described below.

## 3. Critical Findings (Priority-Ranked)

These must be addressed before any predictive pipeline is trusted for live trading.

### CRITICAL-1: Triple-Barrier Label Leakage in P07 (and P08)

**File:** `src/ml/pipeline/p07_combined/evaluator.py:71–145`

`P07Evaluator.run_evaluation()` computes triple-barrier labels on the **full dataset**, then splits 70/30. Labels for any training bar within `tpl_bars` of the split boundary are computed using test-set prices.

With `tpl_hours` up to 96 hours at 15-minute resolution, `tpl_bars = 96 × 4 = 384`. This means the last **384 training samples have their label determined by future (test) prices**. With a typical dataset of 2000–5000 bars, 384 contaminated samples represents 7–19% of the training set.

**Symptom in production:** Backtest Sharpe inflated vs. live. The degree of inflation scales with `tpl_hours`.

```python
# CURRENT (broken):
X_f, y_f = cls.prepare_data(ohlcv, params_with_bars)   # labels computed on full set
split_idx = int(len(X_f) * 0.7)
X_train, y_train = X_f[:split_idx], y_f[:split_idx]    # labels near boundary are contaminated
X_test, y_test = X_f[split_idx:], y_f[split_idx:]

# CORRECT:
split_idx = int(len(X_f) * 0.7)
ohlcv_train = ohlcv.iloc[:split_idx]
ohlcv_test  = ohlcv.iloc[split_idx:]
X_train = build_features(ohlcv_train, ...)
y_train = get_triple_barrier_labels(ohlcv_train, ...)   # labels computed on train segment only
X_test  = build_features(ohlcv_test, ...)
y_test  = get_triple_barrier_labels(ohlcv_test, ...)    # labels computed on test segment only
```

**Fix:** Split the raw OHLCV dataframe first, then compute features and labels independently on each segment. Add a buffer of `tpl_bars` rows between the train and test segments to guarantee zero contamination.

### CRITICAL-2: Blocking `input()` in P01 Pipeline Runner

**File:** `src/ml/pipeline/p01_hmm_lstm/run_pipeline.py:266, 282`

When a critical or optional stage fails and `fail_fast=False`, the pipeline calls `input()` to ask the user whether to continue. This causes the process to hang indefinitely when run from a scheduler, cron job, or any non-interactive context (CI, Docker, systemd service).

```python
# Line 266 — will hang forever in non-interactive mode
response = input(f"\nCritical stage {stage_num} failed. Continue with remaining stages? (y/n): ")
```

**Fix:** Replace `input()` with a `--interactive / --no-interactive` CLI flag. Default should be non-interactive (fail silently or continue based on flag). Interactive prompts have no place in a production pipeline runner.

Additionally, the default `skip_stages = [1, 2]` (lines 476–477 when `--skip-stages` is not provided) silently skips data loading and HMM training without any warning. This must be made explicit — either require the flag or default to running all stages.

### CRITICAL-3: Optuna Validation Set Used for Both Tuning and Reporting

**File:** `src/ml/pipeline/p07_combined/optimize.py:10–52` and `evaluator.py:89–91`

The 30% held-out split inside `run_evaluation()` is used as Optuna's objective signal. After 500 trials, the parameters that maximize performance on that specific 30% window are selected. This is a form of **indirect test-set leakage**: the "test" set is effectively a validation set that has been tuned against.

`save_artifacts()` then re-runs the best parameters on **the same data** and calls the result "final performance" — but there is no true held-out test set at this stage. The only genuine OOS validation is in `run_batch()` when multiple year-files are available, which uses the last file as a validation set. But this is not enforced — single-file runs have no true OOS.

**Fix:** Implement a three-way split: train (60%) / validation for Optuna (20%) / true test reported in artifacts (20%). Never tune hyperparameters against the test set. The robustness checker (`P07RobustnessChecker`) already provides walk-forward analysis — it should be mandatory, not optional.

### HIGH-1: P07 Evaluator Not Using Shared `timeseries_split.py`

**File:** `src/ml/pipeline/p07_combined/evaluator.py:89–91`

The shared module at `src/ml/pipeline/shared/timeseries_split.py` enforces correct split-before-scale and no-shuffle semantics. P07 bypasses it entirely, rolling its own split with a raw slice. All pipelines should use `split_and_scale()` from the shared module. This is primarily an architectural consistency problem, but it also means P07 misses any future safety improvements added to the shared utility.

---

### HIGH-2: Rolling Memory Phase Detection Requires Historical Artifacts

**File:** `src/ml/pipeline/p06_emps2/emps2_pipeline.py:377–380`

The rolling memory scan in Stage 4 reads from `results/p06_emps2/<date>/` directories from prior runs. On first deployment (or after clearing results), there is no historical data and phase detection silently returns empty DataFrames. This is documented in the code comments but not in alerting or operational runbooks.

**Consequence:** On a fresh server, P06/P10 will produce no Phase 1 or Phase 2 alerts for 14 days (the lookback window). There is no alert or health-check to detect this condition.

**Fix:** Add an explicit operational health-check: if rolling memory has been active for more than 14 days with zero Phase 1/Phase 2 candidates ever detected, emit a warning alert. Document the bootstrap period in the ops runbook.

---

### HIGH-3: Robustness Checks Are Optional and Manual

**Files:** `p07_combined/pipeline.py`, `p07_combined/run_robustness_checks.py`

`P07RobustnessChecker` implements walk-forward analysis (5 windows), Monte Carlo shuffles, and parameter sensitivity. These are **excellent** validation tools. However, they must be invoked manually via a separate script. Nothing in the main pipeline flow requires robustness checks to pass before a model is declared "completed" (marked with `completed.flag`).

**Fix:** Run the robustness suite automatically after artifact generation. Gate the `completed.flag` on a minimum walk-forward efficiency ratio (e.g., avg OOS Sharpe / IS Sharpe ≥ 0.5). Reject and re-tune if the strategy fails.

---

### HIGH-4: TRF Data Lag Inconsistency Risk

**Files:** `p04_short_squeeze`, `p06_emps2`, `p10_emps3`

TRF (FINRA short-sale volume) data is published with approximately a 2-week lag. The pipelines download TRF for a target date and apply it as a correction factor to intraday yfinance volume. The shared `trf_downloader.py` handles the lag correctly for historical bars, but there is a risk that some pipelines cache a TRF correction factor computed from a specific date and re-apply it to future bars without refreshing.

**Fix:** Add an explicit TTL validation to the TRF cache: if the cached file is older than 1 trading day, force a refresh. Log the TRF file date alongside every volume metric computation so the lag is traceable in audit logs.

---

### MEDIUM-1: P09 Arbitrage — Cointegration Estimated on Full History Before Backtest

**File:** `src/ml/pipeline/p09_arbitrage/cointegration_analyzer.py`

The standard Johansen/Engle-Granger cointegration test is typically run on the full available price history, then the spread is traded within the same period. This is **in-sample backtesting**: the cointegration relationship is known to hold because it was estimated on the same data the backtest trades.

A properly specified pairs-trading backtest must:
1. Estimate cointegration on a burn-in window (e.g., first 12–18 months).
2. Trade the spread in the subsequent OOS period.
3. Re-estimate quarterly (cointegration is not stationary).

**Fix:** Add a walk-forward cointegration backtest. Estimate on a rolling 252-day window, trade the next 63-day window. Report the fraction of windows where cointegration held at p < 0.05.

---

### MEDIUM-2: P13 BDSH — No Backtest or Signal Validation

**File:** `src/ml/pipeline/p13_bdsh/`

P13 maintains a `state.json` with live trading state (entry price, current exposure, cooldown flag) and applies VIX-based position sizing. It is listed as "LIVE" but there is no backtest infrastructure, no performance metrics, and no OOS validation evidence. VIX-aware position sizing is not the same as a validated signal — it reduces position size in high-VIX environments but does not predict stock direction.

**Recommendation:** Treat P13 as a **risk overlay module**, not a standalone signal pipeline. Integrate it into P04, P06, or P18 as a position-sizing layer. Do not run it as an independent live strategy.

---

### MEDIUM-3: P01 Indicator Optimization (Stage 4) May Contaminate Labels

**File:** `src/ml/pipeline/p01_hmm_lstm/x_04_optuna_indicators.py`

Indicator parameters (RSI period, Bollinger Band period, etc.) are optimized with Optuna using the same dataset that will be used to train the LSTM. If the indicator optimization objective uses any part of the held-out validation set, then the validation set is contaminated before LSTM training begins. The code must be audited to confirm that Stage 4's Optuna study uses ONLY the training fold.

**Fix:** Enforce a strict split: indicator optimization is allowed to see only the first 60% of data. The LSTM validation (Stage 6) uses the next 20%. The final 20% is a true test set touched only by Stage 7.

## 4. Shared Infrastructure Analysis

### 4.1 `shared/timeseries_split.py`

Design is correct and well-documented. The `split_and_scale()` function enforces:
- Split before scale (prevents preprocessing leakage)
- `shuffle=False` (preserves temporal causality)
- Train-only scaler fitting

**Problem:** Adoption is inconsistent. P07 bypasses it; other pipelines may not use it at all. Shared utilities have zero value unless they are the only path.

**Recommendation:** Audit every pipeline's training loop for direct `train_test_split` or manual slicing calls. Replace all with `split_and_scale()`. Fail CI if `train_test_split(shuffle=True)` appears anywhere in `src/ml/`.

### 4.2 `shared/volatility_filter.py`

The phase detection logic (Accumulation → Breakout → Entry) is the core edge of P06/P10. It uses:
- ATR-based volatility expansion
- Volume z-score above lookback average
- Price close relative to recent high

This is a solid microstructure signal. Key risk: the `vol_zscore` threshold and ATR period are configurable but not separately validated per asset class. Small-cap stocks have different volatility regimes than large-caps; a single threshold may be too permissive for large-caps and too tight for micro-caps.

### 4.3 `shared/trf_downloader.py`

Correctly handles the FINRA 2-week publication lag. The pipeline summary explicitly tracks whether TRF data was available and adjusts volume metrics accordingly. This is a genuine informational edge.

### 4.4 `shared/sentiment_filter.py`

Social sentiment (StockTwits, Reddit) is collected asynchronously with timeout handling. The code correctly degrades gracefully when APIs are unavailable. However, sentiment is collected only for Phase 2 candidates and used only for alerting context — it does not gate whether a ticker is alerted. This is the right design (sentiment is noisy; using it as a hard filter would reduce alert recall).

## 5. Pipeline-by-Pipeline Analysis

### P00: HMM + 3×LSTM (Regime-Aware Feature Extraction)

**Purpose:** Trains three separate LSTMs, one per HMM-detected market regime (bull/bear/sideways). The per-regime specialization prevents a single model from averaging across contradictory patterns.

**Trading correctness:** Sound. The regime-aware architecture is theoretically justified — a bull-market LSTM learns trend-following patterns that would harm performance if applied in a bear regime.

**Risks:**
- If regime detection accuracy is below ~70%, the LSTM receives noisy regime labels, degrading performance.
- Three independently trained LSTMs require 3× the data per regime; rare regimes (e.g., "crash") may have insufficient training samples.
- No shared codebase with P01 despite similar architecture. Diverged implementations increase maintenance burden.

**Recommendation:** Merge P00 and P01 into a single configurable pipeline. P00's multi-LSTM approach is P01 without Optuna + with per-regime training; they should share the same LSTM training and validation code.

---

### P01: HMM-LSTM Pipeline (Production-Grade Orchestrator)

**Purpose:** Full 7-stage pipeline with Optuna for both indicator tuning (Stage 4) and LSTM hyperparameter tuning (Stage 5).

**Architecture quality:** The `PipelineRunner` class is well-designed: stage isolation, critical/optional classification, skip-stage support. The `completed.flag` pattern allows safe resume after partial failures.

**Code-level issues:**
1. Default `skip_stages = [1, 2]` (lines 476–477): silently skips data loading and HMM training. This works if you have cached data, but fails silently on a fresh system and produces confusing errors.
2. `input()` calls block non-interactive execution (see CRITICAL-2).
3. Two Optuna studies (indicators + LSTM) create a 2-dimensional hyperparameter search surface. With 500 trials each, there is high risk of fitting to validation set noise. See CRITICAL-3 for the fix.

**Specific recommendation:** Stage 4 (indicator optimization) and Stage 5 (LSTM optimization) should be combined into a single Optuna study that jointly optimizes indicators + LSTM hyperparameters. This halves the tuning surface area and prevents the sub-optimal scenario where indicator parameters are locked before LSTM tuning begins.

---

### P02: CNN-LSTM → XGBoost (Deep Feature Extraction + Gradient Boosting)

**Purpose:** CNN-LSTM trained as a feature extractor; its hidden-state embeddings are fed to XGBoost for final classification.

**Trading correctness:** Sound in theory. Deep networks capture sequential patterns that XGBoost cannot; XGBoost's decision-tree structure resists the overfitting that pure deep networks suffer from on small financial datasets.

**Critical risk:** Two nested optimization stages (CNN-LSTM Optuna + XGBoost Optuna) compound leakage risk. If the CNN-LSTM overfits to training data, its embeddings encode noise rather than signal. XGBoost then optimizes on noisy embeddings, producing parameters that appear to work in-sample but fail out-of-sample.

**Validation requirement (currently missing):**
- Step 1: Train CNN-LSTM on training fold, extract embeddings for validation fold.
- Step 2: Verify that embeddings on validation fold are predictive (information coefficient > 0.05).
- Step 3: ONLY if step 2 passes, proceed to train XGBoost on embeddings.
- Step 4: Report final metrics on a separate test fold that XGBoost never saw.

**Recommendation:** This pipeline has the highest overfitting risk of all 19. Do not use in live trading without walk-forward validation showing positive OOS Sharpe across at least 3 contiguous OOS windows.

---

### P03: CNN → XGBoost (Simplified Hybrid)

**Purpose:** Single CNN for feature extraction (no LSTM temporal component) → XGBoost.

**Compared to P02:** Fewer parameters, faster training, more interpretable. The CNN captures local price patterns (shapes, candle formations) but misses long-range dependencies that LSTM provides.

**Recommendation:** P03 is a reasonable baseline to validate before running P02. If P03 shows no OOS edge, P02 will not either — the LSTM adds complexity but not information when the underlying signal is weak.

---

### P04: Short Squeeze Detector

**Purpose:** Multi-signal short squeeze screening using FINRA TRF (short-sale volume) + intraday volume pattern detection.

**Trading correctness:** Strong. The combination of structural supply-side data (FINRA short interest, days-to-cover, borrow fee) with demand-side microstructure (volume z-score, ATR expansion) is a theoretically justified approach. FINRA TRF data gives edge because it reflects actual short positioning, not just price action.

**Data leakage:** None. FINRA TRF has a 2-week publication lag; the pipeline uses historical data only.

**Production risks:**
- Short squeeze setups can persist for days before triggering, making entry timing critical. The pipeline detects the setup but does not generate entry signals with specific price/time targets.
- Borrow fee data availability is inconsistent; some brokers don't expose this via API.

**Recommendation:** Add a "days since setup first detected" counter to quantify setup age. Fresh setups (1–3 days in the screener) have different risk/reward than stale ones (10+ days).

---

### P05: EMPS (Explosive Move Probability Score)

**Purpose:** Single-ticker composite score from volume z-score (45%), VWAP deviation (25%), realized volatility ratio (25%), and liquidity (5%).

**Assessment:** P05 is a simplified predecessor to P06/P10. The weights are sensible but the absence of a rolling memory mechanism means P05 cannot distinguish between day 1 of an accumulation phase (low conviction) and day 10 (high conviction). P06/P10 correct this.

**Recommendation:** Deprecate P05. Route all EMPS-type scoring through P06 or P10, which have superior multi-day tracking and TRF volume correction. P05 adds maintenance burden without adding signal quality.

---

### P06: EMPS2 (Enhanced Explosive Move Pre-Screener)

**Purpose:** 8-stage production screener: universe (8000 tickers) → fundamental filter (~500) → volatility/phase filter (~50) → rolling memory (14-day) → UOA analysis → results → sentiment collection → alerts.

**Architecture:** Exemplary. The `EMPS2Pipeline` class is the best-structured pipeline in the codebase: stage isolation, per-scan file logging with cleanup, graceful degradation on API failures, configurable checkpointing.

**Code-level observation:** `target_date` defaults to yesterday, which is correct for end-of-day data. The pipeline correctly separates the _target trading day_ (the data's date) from the _scan timestamp_ (when the pipeline ran). This distinction is critical for backtesting reproducibility.

**One improvement opportunity:** Stage 7 (sentiment) is sequential within the main pipeline run. Since sentiment fetching involves network I/O for potentially 20–50 tickers with timeouts, it could be parallelized with `asyncio.gather()`. The `SentimentFilter` already uses `async` internally; the pipeline stage wrapper just needs to run it concurrently.

---

### P07: Combined (Macro Regime + XGBoost Signal)

**Purpose:** HMM trained on VIX + BTC market cap to detect macro regime, which is injected as a feature into XGBoost trained on per-ticker OHLCV with triple-barrier labels.

**Architecture:** The cross-file validation in `run_batch()` is well-designed: optimization uses all files except the latest, which is held for OOS validation. The Optuna study database (SQLite) enables auditable experiment tracking.

**Critical issue:** See CRITICAL-1 (triple-barrier label leakage at split boundary) and CRITICAL-3 (Optuna validation set contamination).

**Additional observation:** `enrich_data()` trains the macro regime model with `anchor_date = df.index.min()`. This correctly prevents the macro HMM from seeing future data relative to the ticker's history. However, the regime model is retrained every time `enrich_data()` is called (once per ticker). For 50 tickers, this means 50 regime model retrains with slightly different anchor dates. The macro regime model should be trained once and cached for the batch run.

**Objective function:** `adjusted_score = sharpe * np.log10(total_trades)` penalizes strategies with few trades. This is correct — a strategy with 2 lucky trades and Sharpe 3.0 is less trustworthy than one with 100 trades and Sharpe 1.5. The `np.log10` scale is appropriate.

---

### P08: MTF (Multi-Timeframe Extension of P07)

**Architecture:** Extension of P07 with multi-timeframe feature fusion. Longer timeframes provide trend context; shorter timeframes refine entry timing.

**Risks:** Inherits all of P07's issues (CRITICAL-1, CRITICAL-3). Additionally, multi-timeframe data alignment introduces complexity: a 15-minute bar that crosses a daily bar boundary must be handled carefully to avoid look-ahead on the daily bar's close.

**Recommendation:** Fix P07 first. P08 is architecturally P07 with an added data dimension. Any fix applied to P07's evaluator must be applied identically to P08.

---

### P09: Arbitrage (Pairs Trading via Cointegration)

**Purpose:** Identify cointegrated pairs, trade the mean-reverting spread.

**Critical issue:** See MEDIUM-1. Cointegration is estimated on the full history, then spread signals are generated over the same period. This is in-sample backtesting.

**Additional concerns:**
- Pairs trading requires continuous monitoring and tight execution. The `run_backtrader_p09.py` backtester simulates this, but real execution requires low-latency infrastructure not currently present in the platform.
- Cointegration is regime-dependent. A pair that cointegrates during 2020–2022 may not cointegrate during 2023–2025 if the relationship is driven by sector rotation or macro factors that change.

**Recommendation:** Treat P09 as research-only. Walk-forward cointegration validation is required before any live use.

### P10: EMPS3 (EMPS2 with AccumulationAnalyzer)

**Purpose:** Evolution of P06 replacing the volatility filter with a more sophisticated `AccumulationAnalyzer` that tracks multi-day OHL patterns.

**Assessment:** The AccumulationAnalyzer's conviction metrics (vol_delta z-score, intensity relative to ATR) are a genuine improvement over P06's simpler phase logic. However, P10 and P06 are structurally near-identical and have diverged implementation-wise.

**Recommendation:** Merge P06 and P10 into a single configurable screener where the volatility filter is a pluggable component (P06 `VolatilityFilter` vs P10 `AccumulationAnalyzer`). Maintaining two near-identical pipelines doubles maintenance overhead for every shared change (TRF downloader, rolling memory, sentiment, alert sending).

### P12: Order Flow Microstructure

**Purpose:** Detect order flow imbalances (buy/sell pressure, crowded positions) from Level 2 data.

**Assessment:** Theoretically excellent — order flow is the most direct leading indicator available. However, this pipeline is **data-limited**: Level 2 order book data requires a paid data subscription (Interactive Brokers, Alpaca, Polygon.io premium). Without confirmed data availability, this pipeline cannot run in production.

**Recommendation:** Gate the pipeline on data source availability. Add a startup check that validates L2 data access and fails with a clear error if unavailable. Document the required data subscription in `Requirements.md`.

### P13: BDSH (VIX-Aware Position Sizing)

See MEDIUM-2 in Critical Findings. P13 is a risk overlay, not a signal pipeline. Reclassify and integrate.

### P14: ATH (All-Time High Momentum/Reversion)

**Purpose:** Buy at post-ATH troughs, sell at next ATH. Pure mechanical rule, no ML.

**Assessment:** The backtest logic is straightforward and free of data leakage (all events are historical). However, the edge is regime-specific: this strategy works in secular bull markets (NVDA, TSLA 2020–2021) and fails in choppy or bear markets. The pipeline has no regime filter.

**Recommendation:** Add a regime gate: only enter ATH-trough trades when the macro HMM (from P07) detects a bull regime. Without this, the strategy will trade through bear markets and produce large drawdowns.

### P15: Hidden Dependencies (Data Downloader)

**Purpose:** Nightly downloader for all data sources: yfinance OHLCV, CBOE P/C ratio, options chains, CNN Fear & Greed, GDELT, FRED, SEC EDGAR, FINRA TRF.

**Assessment:** This is the data foundation for the entire platform. The self-healing gap detection (backtills up to 60 calendar days) and per-source error handling are production-quality.

**Key risk:** GDELT data quality is highly variable. GDELT's event coding introduces substantial noise; without an explicit quality filter (e.g., CAMEo event relevance score, gold-standard entity extraction), raw GDELT ingestion will produce noisy features.

**Recommendation:** If GDELT-derived features are used in any pipeline, add a minimum relevance score filter at ingestion time. Log GDELT coverage gaps separately from OHLCV gaps — they have different causes and remedies.

---

### P16: Taleb (Options Pricing and Volatility Trading)

**Purpose:** Black-Scholes and stochastic volatility models for options mispricing detection.

**Assessment:** Theoretically sophisticated. However, options pricing models require calibration to a liquid volatility surface. For small-cap stocks (which are the primary target of P04, P06, P10, P17), options are illiquid and bid-ask spreads are wide enough to eliminate any theoretical mispricing edge.

**Recommendation:** Scope P16 to large-cap stocks and ETFs with liquid options markets (SPY, QQQ, individual names with >10K OI). Validate that theoretical mispricings identified exceed the typical bid-ask spread before treating them as tradeable signals.

---

### P17: Penny Stock Screener (7-Agent Pipeline)

**Purpose:** Multi-agent screener for penny stocks (<$5) with 7 isolated agents: universe, market data, technical, short squeeze, dilution risk, scoring, and reporting.

**Assessment:** The agent architecture is well-designed for testability and independent development. The dilution risk detection (tracking shares outstanding growth from filings) is a unique and important edge — dilution is the primary mechanism by which penny stocks destroy shareholder value.

**Production risks:**
- Penny stocks are highly illiquid. A screener alert may cause the reader to try to buy a stock with $50K/day volume, making them the marginal buyer and moving the price against themselves.
- The scoring weights across 7 agents (if configurable) are subject to overfitting.

**Recommendation:** Add a maximum daily dollar volume filter ($500K minimum) to ensure screened stocks are actually tradeable. Add explicit position sizing guidance: maximum position = min(1% NAV, 5% of daily dollar volume).

---

### P18: Institutional Flow Tracker

**Purpose:** Daily pipeline tracking 13F filings (quarterly consensus), Form 4 insider activity, and volume anomalies. Three signal layers combined into a composite score.

**Architecture:** The cleanest pipeline in the codebase. Clean separation of concerns: `EdgarDownloader`, `PositionDeltaCalculator`, `ExitScreener`, `ConsensusDetector`, `VolumeAnomalyDetector`, `Form4Monitor`, `CompositeScorer`. Each component is independently testable.

**Data handling:** Correct use of the 45-day 13F publication lag and 2-day Form 4 lag. The quarterly consensus cache (`consensus/{year}_Q{quarter}.csv.gz`) is updated incrementally as new filings arrive — no need to reprocess the entire quarter on non-filing days.

**Minor code observation:** `_collect_watchlist_tickers()` uses a manual O(n²) deduplication loop (list membership check inside a loop). For the expected list sizes (< 200 tickers) this is harmless, but `dict.fromkeys()` or `list(dict.fromkeys(tickers))` would be idiomatic.

```python
# Current (O(n²) for large lists):
seen: List[str] = []
for t in tickers:
    if t and t not in seen:
        seen.append(t)

# Better:
seen = list(dict.fromkeys(t for t in tickers if t))
```

**One architectural gap:** The composite scorer's weights (`signal_weights` in `P18Config`) are fixed at configuration time. There is no feedback loop: if insider Form 4 sells have been predictive in the past quarter while 13F consensus has been noisy, the weights do not adapt. Consider adding a quarterly weight recalibration step based on the previous quarter's realized signal accuracy.

## 6. Architectural Patterns — What's Working

### 6.1 Stage Isolation with Critical/Optional Classification

P01's `PipelineRunner` pattern (critical/optional stage classification, skip-stage support, `completed.flag`) should be the standard for all predictive model pipelines. P06 already implements a similar multi-stage pattern but without the skip-stage mechanism.

### 6.2 Macro Regime as Context Feature (P07)

Injecting the macro HMM regime (VIX + BTC) as a feature column before XGBoost training is the right approach. It allows the model to learn regime-conditional patterns without requiring separate per-regime models. This approach should be propagated to P01, P02, P03.

### 6.3 FINRA TRF Volume Correction

Correcting yfinance intraday volume with FINRA dark pool short-sale data is a genuine informational edge. The shared `trf_downloader.py` handles this correctly. This is used in P04, P06, P10, P17 — it should also be considered for P12 (order flow).

### 6.4 Robustness Suite in P07

`P07RobustnessChecker` implements walk-forward analysis, Monte Carlo trade shuffling, and parameter sensitivity testing. These are the right validation tools. The problem is they are optional. See HIGH-3.

### 6.5 Optuna Study Persistence (SQLite)

Persisting Optuna studies to SQLite enables re-use across runs, experiment comparison, and audit trail. This is good practice. All Optuna-using pipelines should use this pattern.

## 7. Architectural Patterns — What Needs to Change

### 7.1 No Unified Backtest Framework

Different pipelines use different backtest tools: vectorbt (P07, P08), Backtrader (P09), custom equity curves (P13, P14). This makes cross-pipeline performance comparison impossible. Standardize on one framework — vectorbt is already present in the more mature pipelines and offers the best performance for large-scale backtesting.

### 7.2 No Centralized Model Registry

Trained models are saved to local directories (`results/{pipeline}/{ticker}/{timeframe}/`). There is no model registry, versioning, or performance tracking over time. When a model is retrained, the previous version's metrics are overwritten (though the `.json` artifacts remain). MLflow integration exists in `src/ml/mlflow_integration.py` but is not used by any pipeline.

**Recommendation:** Either adopt MLflow for model tracking, or build a minimal custom registry: a `models.sqlite` database with columns (pipeline, ticker, timeframe, train_date, oos_sharpe, n_trades, artifact_path). Query this to compare model versions and detect performance drift.

### 7.3 Pipeline Duplication

Several pipelines are near-duplicates of predecessors:
- P05 → P06 → P10: same screener with increasing sophistication. Only P10 should be maintained.
- P00 → P01: same HMM+LSTM architecture, P01 adds Optuna. Merge.
- P03 → P02: P03 is P02 without LSTM. Keep both as configuration variants of one pipeline.
- P07 → P08: P08 is P07 with multi-timeframe data loading. Merge.

Consolidating to ~12 distinct pipelines (from 19) would reduce maintenance burden by 37% with no loss of signal coverage.

### 7.4 No Live Signal → Trade Pipeline Bridge

All screener pipelines (P04, P06, P10, P17, P18) produce lists of tickers with signal scores. There is no standardized interface between "signal generated" and "order submitted." Each pipeline sends alerts independently. When two pipelines alert on the same ticker (e.g., P06 detects accumulation AND P18 detects institutional buying), there is no mechanism to combine the signals into a stronger conviction trade, or to avoid position duplication.

**Recommendation:** Define a `Signal` dataclass with fields: `ticker`, `pipeline_id`, `signal_date`, `signal_type`, `score`, `confidence`, `metadata`. All pipelines emit `Signal` objects to a central `SignalBus`. The execution layer reads from `SignalBus` and handles deduplication, position sizing, and order routing.
