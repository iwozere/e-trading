## Critical Bugs (Breaking Results)

### 1. P06/P10 — Rolling Memory Always Blind in Accumulation Mode
**File:** `src/ml/pipeline/p06_emps2/rolling_memory.py:95`

```python
vol_file = day_folder / '05_volatility_filtered.csv'  # hardcoded!
```

But `AccumulationAnalyzer._save_results()` writes to:
```python
out = self._results_dir / "07_prebreakout_watchlist.csv"  # different name!
```

When `analyzer_type='accumulation'` (P10 uses this exclusively), every historical scan result is saved as `07_prebreakout_watchlist.csv`, but rolling memory only looks for `05_volatility_filtered.csv`. **Rolling memory never finds any historical data → Phase 1 and Phase 2 are always empty → no alerts are ever sent.** This is the most likely reason P10 produces no results.

**Fix needed:** Rolling memory scanner must also check for `07_prebreakout_watchlist.csv` (or the filename must be standardized).

---

### 2. P18 — 13D/G Score Applied to Every Ticker (Not Just Matched Ones)
**File:** `src/ml/pipeline/p18_institutional_flow_tracker/scoring/composite_scorer.py:131`

```python
# --- Schedule 13D/G drop ---
if not dg_df.empty:             # checks if ANY 13D/G exists in the whole dataset
    detail["schedule_13dg_drop"] = True
    score += self._weights["schedule_13dg_drop"]  # +10 applied to EVERY ticker
```

The `if not dg_df.empty` check is supposed to check whether *this ticker* has a 13D/G event, but it actually checks whether the entire `dg_df` DataFrame has any rows at all. On any day where any institution files a 13D/G on any stock, **all tickers get +10 points**. This inflates scores for unrelated tickers and corrupts the composite signal.

**Fix needed:** Replace with `if not dg_df[dg_df["ticker"] == ticker].empty`.

### 3. P18 Reader — Looking for Insider Buys in the Sells File
**File:** `src/ml/pipeline/p05_ai_selector/signals/p18_reader.py:78-83`

```python
form4_file = run_dir / "form4_sells.csv"   # file contains SELLS
if form4_file.exists():
    df = pd.read_csv(form4_file)
    buys = df[df["transaction_type"] == "B"]  # looking for BUYS inside a SELLS file
    result["form4_buy_tickers"] = set(buys["ticker"]...)
```

`Form4Monitor.get_significant_sells()` writes a file called `form4_sells.csv` containing only sell transactions. Searching it for `transaction_type == "B"` will always return an empty set. The P05 stage 2 `form4_insider_buy` signal boost (+15 points) is dead code.

### 4. P06/P10 — Vol Z-Score Includes Today's Bar in Its Own Baseline
**File:** `src/ml/pipeline/p06_emps2/accumulation_analyzer.py:347-354`

```python
def _compute_zscore(self, series: np.ndarray, window: int) -> float:
    mean = np.mean(series[-window:])   # includes today's bar
    std  = np.std(series[-window:])    # includes today's bar
    return (series[-1] - mean) / std
```

Today's volume bar is included in both the mean and std used to evaluate itself. On a genuine accumulation day with high volume, including it in the baseline inflates the mean and std, *suppressing* the z-score. The filter then rejects real accumulation signals (`vol_zscore <= min_vol_zscore = 1.5`).

**Fix needed:** Use `series[-(window+1):-1]` for the baseline, then evaluate `series[-1]` against it.

### 6. P17 — ThreadPoolExecutor Has Only 1 Worker (No Parallelism)
**File:** `src/ml/pipeline/p17_penny_stocks/agents/universe_agent.py:175`

```python
with ThreadPoolExecutor(max_workers=1) as pool:  # defeats the purpose
```

The entire NASDAQ universe (~8000 tickers) is fetched sequentially. With 1–2 seconds per ticker due to Yahoo rate limiting, this takes 2–4 hours. Increasing `max_workers` to 4–8 with proper rate-limit backoff would cut runtime by 4–8×.

---

### 7. P05 / P18 — Results Path is Relative (CWD-Dependent)
**Files:**
- `src/ml/pipeline/p05_ai_selector/config.py:23` — `P18_RESULTS_BASE = Path("results") / "p18_institutional_flow"`
- `src/ml/pipeline/p18_institutional_flow_tracker/config.py:80` — `results_dir: Path = Path("results") / "p18_institutional_flow"`

Both use relative paths. When the scheduler runs either pipeline from a directory other than the project root (or from `r:\results` on the server), neither will find the other's output. The P18 signal boost in P05 Stage 2 will silently return zeros for every ticker.

**Suggested fix:** Use `Path(__file__).resolve().parents[N] / "results" / ...` or configure absolute paths via the scheduler.

---

### 8. P06 — NASDAQ FTP URLs May Be Dead
**File:** `src/ml/pipeline/p06_emps2/config.py:65-66`

```python
nasdaq_url: str = "ftp://ftp.nasdaqtrader.com/SymbolDirectory/nasdaqlisted.txt"
other_url:  str = "ftp://ftp.nasdaqtrader.com/SymbolDirectory/otherlisted.txt"
```

NASDAQ deprecated their FTP endpoints and now serves these files over HTTPS. If the universe download fails silently (returns empty list), the pipeline logs `"Failed to download universe"` and exits immediately at Stage 1.

**Fix needed:** Switch to HTTPS: `https://www.nasdaqtrader.com/dynamic/SymbolDirectory/nasdaqlisted.txt`

### 10. P06/P10 — `require_dark_pool_surge` Config Field Is a Dead Flag

## Production Log Analysis — 2026-06-23

Reviewed `app.log` from the production server (Raspberry Pi, user `alkotrader`). Findings below are based on the 09:00 AM P18 run and subsequent pipeline runs on 2026-06-22.

### P18 Root Cause Confirmed — EDGAR EFTS API Returning HTTP 500

```
09:00:01 ERROR - EFTS search failed (forms=13F-HR start=2026-06-22 end=2026-06-22)
HTTPError: 500 Server Error: Internal Server Error
  url: https://efts.sec.gov/LATEST/search-index?forms=13F-HR&...

09:00:02 ERROR - EFTS search failed (forms=SC 13D,SC 13G,SC 13D/A,SC 13G/A ...)
HTTPError: 500 Server Error: Internal Server Error
```

EDGAR's full-text search index (`efts.sec.gov`) is returning 500 on both 13F-HR and 13D/G queries. The downloader catches these errors internally and returns empty DataFrames. The pipeline continues with:
- 0 new 13F filings (no consensus built — never was)
- 0 Form 4 sell transactions (also EFTS-based)
- 0 13D/G filings
- Empty volume anomaly watchlist → empty composite score → nothing written to dated directory

The scheduler then logs `"Successfully completed job for P18"` because the outer `run()` method returns a result dict without raising. The dated result directories are empty because all DataFrames are empty and every write is guarded by `if not df.empty`.

**The pipeline is completing successfully from Python's perspective — it has no awareness that all three EDGAR data sources returned zero records due to the upstream 500.**

EDGAR's EFTS endpoint is a separate service from the main filing system and experiences periodic outages. This appears to have been ongoing since at least early June (dates of the empty result directories).

**Code fix required:** The pipeline must detect when all EDGAR sources return zero records on a non-weekend trading day and either:
1. Fall back to the prior day's cached consensus signals, or
2. Raise an explicit alarm rather than silently producing empty output.

### Infrastructure — yfinance Cache Permissions (Process User Mismatch)

```
INFO - Failed to create TzCache folder: '/home/alkotrader/.cache/py-yfinance'
  reason: [Errno 13] Permission denied: '/home/alkotrader'
```

The error is on `/home/alkotrader` itself, not on `.cache/`. The home directory is `drwxr-x---` (mode 750), which means any process not running as `alkotrader` or in the `alkotrader` group cannot enter it. This indicates the trading app runs as a different user (e.g., root inside Docker with a UID mismatch, or a service account). `chown` on the home directory does not fix this — the process needs a writable cache directory it can actually reach.

**Fix:** Redirect yfinance to use `$DATA_CACHE_DIR/yfinance`, which is already writable by whatever user runs the pipelines. Add `yf.set_tz_cache_location()` in `src/data/downloader/yahoo_data_downloader.py` at class initialization using the same `DATA_CACHE_DIR` env var used by all other downloaders.

### Alpaca Subscription Limitation (Ongoing Data Quality Degradation)

```
WARNING - "subscription does not permit querying recent SIP data" (ACLX, KALV, THAR, FFIC, EWCZ, SCVL)
```

The current Alpaca subscription tier does not include real-time SIP (Securities Information Processor) data. Recent OHLCV bars (last 1–3 days) for many tickers fall back to Yahoo Finance or return empty. This degrades P06/P10 signal quality for the most recently active tickers — precisely the candidates the pipeline is designed to find.

**Options:** (1) Upgrade Alpaca to a plan with SIP data, or (2) accept Yahoo Finance as the primary recent-bar source and use Alpaca only for historical backfill.

## EDGAR Backfill (2026-06-23)

**13F-HR quarterly consensus:** The daily pipeline only fetches filings submitted *on the run date*. During the EFTS outage (~early June to 2026-06-22), those days' filings were never processed — they are missing from the quarterly consensus.

`InstitutionalFlowPipeline.rebuild_quarterly_consensus(year, quarter)` was added to recover from this:
1. Re-downloads the full quarter 13F index (all filings in the 50-day filing window).
2. Fetches any infotables not already cached.
3. Rebuilds the consensus from the complete dataset and writes it to the consensus cache.

Call it once after EFTS comes back up:
```python
from src.ml.pipeline.p18_institutional_flow_tracker.pipeline import InstitutionalFlowPipeline
pipe = InstitutionalFlowPipeline()
pipe.rebuild_quarterly_consensus(2026, 2)  # current quarter
```

**Form 4 / 13D/G:** Skipped. The pipeline uses only `yesterday`'s data per run — historical days are not consumed by any downstream logic, so backfilling them adds no ongoing signal value.
