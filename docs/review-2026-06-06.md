## 1. `src/scheduler` (4 files, ~2.1k LOC)

### Architecture assessment

APScheduler-based async service with PostgreSQL job persistence (`SQLAlchemyJobStore`), DB `LISTEN/NOTIFY`-driven hot reload, and five job types (alert, portfolio, screener, report, data-processing). The design is clean and clearly post-review: debounced reloads, atomic reconciliation (`reload_schedules`), per-job timeout via `asyncio.wait_for`, misfire/coalesce tuning, and path-traversal validation for subprocess jobs are all present and well-reasoned. `main.py` does proper DI and signal-safe graceful shutdown.

**The dominant architectural risk is the async/sync boundary.** The service runs jobs on `AsyncIOExecutor` (everything on one event loop), but every database touch goes through **synchronous** services (`JobsService`, `NotificationService`, `UsersService`) called *without* a thread offload. `list_schedules`, `get_schedule`, `create_run`, `update_run`, `update_schedule_next_run` are all blocking calls executed directly inside `async def` methods. A single slow query (or a connection-pool stall) blocks **all** concurrent job evaluations and the DB `LISTEN` heartbeat. With `max_workers`/`max_instances` set to 10, this silently serializes under load.

**Split-readiness:** the module-level `_service_instance` singleton (`scheduler_service.py:44,119-126`) hard-caps the service to **one instance per process**, which prevents horizontal scaling of the scheduler. That is acceptable for a leader-elected singleton deployment, but if the platform is split into sub-apps, the scheduler must remain a single-writer (or move to a distributed scheduler / leader election). Document this constraint explicitly.

## 2. `src/common` (86 files, ~27k LOC)

### Architecture assessment

`common` is doing too much to be a single "shared utilities" module. It contains at least five distinct concerns of very different weight:

1. **Provider/data facade** (`common.py`) ŌĆö thin convenience wrappers over `DataManager`.
2. **Alerts engine** (`alerts/`) ŌĆö `AlertEvaluator` (1.7k LOC), cron parsing, JSON-schema rule validation, and a plugin registry. This is a genuine subsystem.
3. **Sentiment platform** (`sentiments/`, ~50 files) ŌĆö adapters for 10+ providers (Reddit, Twitter, StockTwits, Finnhub, news, trendsŌĆ”), a multi-tier cache (memory + Redis), a rate-limiting subsystem (adaptive limiter, global coordinator, priority queue), bot detection, HF transformer analysis, and performance profiling. **This is an application in its own right**, not a "common" helper.
4. **Recommendation engine** (`recommendation/`) ŌĆö technical + fundamental rule scoring.
5. **Health/heartbeat** (`health_monitor.py`, `heartbeat_manager.py`, `health_cli.py`) ŌĆö cross-cutting observability.

**Split recommendation (high value):** `sentiments/` should become its own top-level package/sub-application (`src/sentiment/`) with its own deployable surface ŌĆö it has independent scaling characteristics (heavy external I/O, its own rate limiting and caching) and would benefit from running as a separate worker. `alerts/` and `recommendation/` are also cohesive enough to be promoted out of `common`. What remains in `common` should be genuinely shared, low-dependency primitives. As-is, importing anything from `common` risks dragging in aiohttp, redis, transformers, psutil, and the entire data layer.

**Quality of the alerts engine:** strong. `_evaluate_rule_tree` / `_evaluate_comparison` use an explicit, whitelisted operator set (`gt/gte/lt/lte/eq/ne/between/outside/crosses_above/crosses_below`) ŌĆö **no `eval`/`exec`**, NaN-guarded operand resolution, and stateful crossing detection. This is the right way to build a user-configurable rule engine.

## 3. `src/data` (184 files, ~55k LOC)

### Architecture assessment

`data` is the platform's backbone and the largest module. It has four well-separated layers:

1. **Persistence (`db/`)** ŌĆö `core` (engine/session/UoW), `models`, `repos`, `services`, `schemas`, Alembic `migrations`. **This is the strongest-engineered part of the codebase.** Lazy thread-safe engine with double-checked locking, `session_scope` context manager, a UoW pattern keyed on a `ContextVar` (correct for both threads *and* asyncio tasks), and an `ALLOW_DROP_TABLES=1` safety gate on destructive ops. The atomic message-claim query (`repo_notification.py:367`) uses `FOR UPDATE SKIP LOCKED` with bound params ŌĆö textbook safe-concurrent-queue design.
2. **Downloaders (`downloader/`, ~25 providers)** ŌĆö clean `BaseDataDownloader` with single responsibility (fetch only; caching/rate-limiting/validation delegated to `DataManager`). API keys resolved through a single cached `provider_config.get_api_key` facade.
3. **Feeds (`feed/`)** ŌĆö live/streaming sources.
4. **Orchestration (`data_manager.py`)** ŌĆö `ProviderSelector` (symbol classification + failover) and `DataManager` (gap-aware OHLCV retrieval, batch, fundamentals, live feeds, per-provider rate limiting).

**Two architectural concerns dominate:**

- **`data_manager.py` is a 3,063-line god-file** holding two large classes with many responsibilities (classification, provider selection, gap detection, caching, fundamentals combination, live-feed factory, rate limiting). It is the single biggest maintainability liability in the module and a magnet for merge conflicts and regressions. It should be decomposed (see DATA-2).
- **Sync-everywhere DB services.** Confirmed at `db/services/base_service.py` ŌĆö `with_uow`/`handle_db_error` are synchronous wrappers. This is fine for the threaded bots, but it is what makes the async callers (scheduler, alert evaluator) block the event loop (SCHED-1/COMMON-2). A clear async story (or a documented "DB calls must be offloaded" rule) is needed before splitting services out.

**Split-readiness:** `db/` is cleanly layered and could be packaged as a standalone data-access library shared by every sub-app. `downloader/` + `feed/` could become a "market-data service." The blocker is `data_manager.py`'s coupling ŌĆö decompose it first.

## 4. `src/notification` (63 files, ~26k LOC)

### Architecture assessment

A genuinely well-built, **DB-centric** notification platform. Core flow: producers write `Message` rows ŌåÆ a `MessageProcessor` claims them atomically (`repo_notification.py:367`, `FOR UPDATE SKIP LOCKED`) ŌåÆ delivers through pluggable channels (email/SMS/telegram) with priority handling, a `FallbackManager`, rate limiting, delivery tracking, archival, analytics, and a health monitor. The processor is correctly async with a `ThreadPoolExecutor` for blocking work and graceful shutdown. Channels are async (`aiosmtplib`) so SMTP doesn't block the loop, use `ssl.create_default_context()`, and **default `validate_certs=True`**.

**The logger (`logger.py`) is platform-critical and notably strong:** multiprocessing-safe `QueueHandler`/`QueueListener`, per-PID child log files to avoid rotation races, UTF-8 enforcement, context-aware handler sharing (not cloning ŌĆö avoids dual rotation state), and a `SensitiveDataFilter` that masks API keys/tokens. This is well above average.

**Split-readiness: excellent.** This module is the cleanest standalone-service candidate in the platform ŌĆö it already operates as a DB-queue worker (`notification_db_centric_bot.py`), so producers need only write rows. The one coupling to sever is its direct dependency on `src.data.db.models` / repos: extract a thin shared "notification schema/contract" package so the service and its producers don't both reach into the data layer's internals.

## 5. `src/trading` (69 files, ~18k LOC) ŌĆö highest-stakes module

### Architecture assessment

The trading module has a thoughtful structure: an abstract `BaseBroker` with a unified order/position/portfolio model and seamless paper/live switching; concrete `BinanceBroker`/`IBKRBroker`/`MockBroker`; a `RiskController` split cleanly into pre-trade / real-time / post-trade phases; a thread-safe `ExecutionPersistenceService`; and a `base_trading_bot` orchestrator that runs pre-trade risk gates before placing orders. The paper-trading simulation (slippage models, latency, partial fills, execution-quality scoring) is unusually rich.

**However, this is also where the most serious defects live ŌĆö and in a trading system these are the ones that cost money.** Two are effectively latent P1s on the live path, and one is a dimensional error in the risk-sizing math. Details below.

`BaseBroker` (1,724 LOC) is overloaded: it mixes the live broker interface, the entire paper-trading simulation engine, *and* the backtrader adapter layer in one class. These are three separable concerns (live execution / simulation / framework bridge) and should be split ŌĆö both for clarity and so the live path can be reviewed and tested in isolation from simulation code.

**Split-readiness:** `trading` is a strong standalone sub-application candidate (it already has `raspberry_pi_trading_service.py`, `trading_runner.py`, `instance_service.py`, `services/trading_bot_service.py`). Before splitting, the live broker layer needs the correctness fixes below and a clear async story (it currently mixes async method signatures with synchronous blocking SDK calls).

## Cross-cutting themes

These patterns recur across modules and deserve platform-level decisions rather than per-file fixes.

### A. The async/sync boundary is the platform's biggest systemic risk
Services that own the data layer are **synchronous** (`db/services/base_service.py`), yet several long-running components run on an `asyncio` event loop and call them inline:
- `scheduler` (SCHED-1) ŌĆö blocks all jobs + the DB listener.
- `alerts/alert_evaluator` (COMMON-2) ŌĆö blocks every concurrent evaluation.
- `trading` Binance broker (TRAD-4) ŌĆö blocks the loop during live order I/O.
- `telegram` business logic (TG-3) ŌĆö blocks command handling for all users.

`notification` is the model done right (async channels + `ThreadPoolExecutor` for blocking work). **Decide one platform rule**: either (a) provide async service variants, or (b) mandate `asyncio.to_thread` at every asyncŌåÆsync DB/SDK call site and lint for it. Until then, throughput claims (`max_workers=10`, etc.) are illusory.

### B. Live-trading correctness needs a dedicated hardening pass
`trading` carries the only money-losing defects: a latent crash on partial-fill status (TRAD-1), an inoperative risk-size cap from a units/dollars mismatch (TRAD-2), unsafe order placement with no post-timeout reconciliation (TRAD-3), and zeroed live position/cost-basis feeding risk gates (TRAD-7). These should be fixed and covered by tests **before** any further live use.

### C. God-files concentrate risk
`data_manager.py` (3,063), `base_broker.py` (1,724), and `alert_evaluator.py` (1,696) hold multiple responsibilities each. They are the hardest files to test and the most regression-prone. Decompose along the seams noted per-module.

### D. Secrets & logging hygiene is good but over-reliant on a regex
The `SensitiveDataFilter` + central `provider_config.get_api_key` facade are strong. But DEBUG-by-default logging (NOTIF-1) plus partial masking coverage (NOTIF-2) means sensitive payloads can still reach logs. Default to `INFO` in prod and keep secrets/PII out of log arguments at the source.

### E. "common" and "telegram" host code that belongs elsewhere
`common/sentiments` (a full platform) and `telegram/screener` (reusable business logic) are mis-located. Moving them is the highest-leverage refactor for split-readiness.

## 8. `src/api` (23 files, ~7.6k LOC)

### Architecture assessment

A well-secured FastAPI backend for the web UI ŌĆö the strongest security posture in the codebase. Highlights:
- **Passwords:** bcrypt (`model_users.py` `set_password`/`verify_password`).

## 11. `src/ml` (179 files, ~49k LOC) ŌĆö by far the largest module

`ml` is an offline research/training estate: **17 numbered pipelines** (`p00`ŌĆō`p17`: HMM-LSTM, CNN-XGBoost, short-squeeze, EMPS variants, MTF, arbitrage, meta, order-flow, ATH, Taleb, penny-stocksŌĆ”), a `pipeline/shared` area, and a `future/` experimental zone. Each pipeline follows an `x_0N_*` step convention (download ŌåÆ feature-eng ŌåÆ train ŌåÆ validate ŌåÆ export).

**The good news on the metric that matters most for financial ML ŌĆö temporal data leakage:** the production pipelines I sampled (`p01_hmm_lstm/x_05_optuna_lstm.py`, `p02_cnn_lstm_xgboost/x_02_feature_engineering.py`) use `train_test_split(..., shuffle=False)` with explicit "Don't shuffle time series" comments, and `p00` does too. That is the correct discipline and shows real awareness.

**The bad news:** that discipline is **not uniform**. `future/automated_training_pipeline.py:83` uses `train_test_split` with the sklearn default (`shuffle=True`) on data that is already `X_scaled` ŌĆö i.e. both **temporal leakage** (shuffled split) and **scaler leakage** (scaler fit on the full set before splitting). It's in `future/`, but it illustrates the central risk of 17 hand-rolled pipelines: each re-implements feature engineering / scaling / splitting, so leakage-correctness must be re-verified per pipeline rather than guaranteed by a shared, audited primitive.

**Split-readiness:** `ml` is unambiguously its own **offline sub-application** (training/optimization). Only the *inference* path ŌĆö model loading in `strategy/cnn_xgboost_strategy.py`, `strategy/hmm_lstm_strategy.py`, and the HMM-LSTM entry mixin ŌĆö couples to the runtime. That inference boundary should be a small, well-defined model-loading interface; the training estate should not be importable from the live trading path at all.

## Pass 2 ŌĆö cross-cutting themes (extends pass 1)

- **Theme A (async/sync) recurs** in `api` (API-3) and the scheduler-driven `portfolio.pnl_alert` path (BT-3). Same platform-level decision applies.
- **New theme F ŌĆö duplication of cross-cutting logic.** The same capability is implemented in multiple places: two TA-Lib adapter layers (SHARED-1), two backtest engines (BT-4), screeners in both `util` and `screeners` (UTIL-1), analytics in three places (ANALYTICS-1), and two `indicator_factory` modules (IND-2). Each duplication is a future correctness-divergence bug. Establish single owners for: indicator computation, backtest engine, screening, and analytics.
- **New theme G ŌĆö config-falsy bugs.** The `x or default` idiom (STRAT-1, 92 sites) silently discards legitimate `False`/`0` values. Worth a repo-wide lint for `get(...) or` / `get_param(...) or` patterns on config reads.
- **ML leakage discipline is good but not enforced** ŌĆö centralize leakage-safe primitives (ML-1/ML-2) so correctness isn't re-litigated per pipeline.
- **Security posture is strong overall** ŌĆö `api` (JWT/bcrypt/RBAC/rate-limit/CORS-safe-default/localhost+HMAC internal routes) and `config` (env-first, fail-closed) are well done; the remaining items are hardening (API-1/2/4), not holes.
